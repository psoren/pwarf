# 16 — Cave System

> **Status:** Implemented
> **Last verified:** 2026-03-26

## Overview

Each fortress has up to 5 cave entrances on the surface, each leading to an independent underground cave at a unique negative z-level. Caves are procedurally generated from the world seed and tracked in the `caves` database table for explicit discovery state and terrain-derived properties.

## Cave Generation

### Entrance Placement

Entrances are placed by `pickCaveEntrances()` in `shared/src/fortress-gen-helpers.ts`:

1. Sample noise on a 32-tile grid across the 512x512 fortress
2. Keep candidates with noise value > 0.7
3. Sort by noise value (highest first)
4. Select up to 5 positions with minimum 80 Manhattan distance between them
5. Sort final positions top-to-bottom, left-to-right

Each entrance gets z-level `-(index + 1)`: first entrance is z=-1, second z=-2, etc. The z-level is stored in the `caves` table so it can be changed independently of generation order in the future.

### Cave Layout (Cellular Automata)

Each entrance gets its own 128x128 cave grid, offset at (192, 192) within the 512x512 fortress coordinate space. The grid is generated by `buildCaveForEntrance()`:

1. **Initial noise fill** — each cell is open with probability `CA_INITIAL_OPEN` (0.45)
2. **Smoothing** — 5 iterations of cellular automata (cell becomes solid if >=5 of 8 neighbors are solid)
3. **Region culling** — open regions smaller than 50 tiles are filled in
4. **Tunnel connection** — if multiple disconnected regions survive, L-shaped tunnels connect them
5. **Entrance guarantee** — the center 3x3 area is always open

### Materials

Ore and gem deposits are placed via separate noise layers per cave, checked by `checkCaveMaterial()`:

| Material | Kind | Noise Threshold |
|----------|------|----------------|
| Iron     | ore  | 0.94           |
| Copper   | ore  | 0.94           |
| Tin      | ore  | 0.95           |
| Gold     | ore  | 0.97           |
| Silver   | ore  | 0.96           |
| Ruby     | gem  | 0.97           |
| Sapphire | gem  | 0.97           |

Cave mushrooms spawn on open floor tiles where mushroom noise > 0.78.

### Cave Names

Generated deterministically from a per-entrance seed: `"The {adjective} {noun} of {material}"`.

## Caves Table

Cave rows are pre-created at embark time with `discovered = false`. Schema:

| Column | Type | Purpose |
|--------|------|---------|
| `id` | uuid | Primary key |
| `civilization_id` | uuid | FK to civilizations |
| `entrance_x`, `entrance_y` | int | Surface entrance position |
| `z_level` | int (-19 to -1) | Underground z-level for this cave |
| `name` | text | Procedurally generated cave name |
| `discovered` | boolean | Set to true when scouted |
| `discovered_by` | uuid | FK to the dwarf who scouted it |
| `discovered_at` | timestamptz | When discovery happened |
| `danger_level` | int (0-100) | Terrain-derived base danger |
| `resource_type` | text | Terrain-derived resource bonus (ore, gem, mushroom, or null) |

### Terrain-Derived Properties

Set at embark based on the overworld terrain type:

| Terrain  | danger_level | resource_type |
|----------|-------------|---------------|
| mountain | 40          | ore           |
| forest   | 20          | mushroom      |
| plains   | 10          | null          |
| desert   | 30          | null          |
| tundra   | 20          | null          |
| swamp    | 60          | mushroom      |
| volcano  | 80          | gem           |

`danger_level` and `resource_type` are not yet used by game logic — they exist for future features.

## Discovery Flow

1. Player clicks a `cave_entrance` tile on the surface
2. If not yet scouted, `CaveScoutModal` appears, player can send a dwarf
3. Dwarf walks to entrance, works for `WORK_SCOUT_CAVE` (50) ticks
4. `completeScoutCave()` runs:
   - Writes a `cavern_floor` marker tile at the entrance position in the cave z-level
   - Sets `cave.discovered = true`, `discovered_by`, `discovered_at`
   - Fires a `discovery` category world event
5. Cave appears in the **Locations** tab in the right panel
6. Clicking the cave entrance again enters the cave (z-level switch + viewport center)

## FortressDeriver Integration

`createFortressDeriver()` accepts an optional `knownCaves` parameter:

- **With knownCaves** (production): Uses DB z-levels and names instead of computing from index
- **Without knownCaves** (tests): Falls back to index-based z-level assignment

The deriver does not store any state — it derives tiles purely from noise functions. Only player-caused changes (mining, building) are stored as `fortress_tiles` overrides in the database.
